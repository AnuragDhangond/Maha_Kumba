Maha Kumbh Mela - Database Architecture & System Flow

1. Database Tables & Logical Modules

Based on the Spring Boot backend (@Entity and @ElementCollection), the application automatically generates 29 tables. They are grouped into the following logical modules:

A. User & Authentication Module

  • users: Central user table storing credentials, roles, and status.
  • access_tokens: Stores short-lived JWT tokens. Relationship: OneToOne with users.
  • refresh_tokens: Stores long-lived tokens for session renewal. Relationship: OneToOne with users.
  • user_activity: Audit table logging user IP, visited pages, and session duration.

B. E-commerce & Souvenir Shop

  • products: Inventory of items available for purchase (price, stock).
  • carts: Active shopping cart for a user. Relationship: OneToOne with users.
  • cart_items: Line items inside a cart. Relationship: ManyToOne with carts & products.
  • shop_orders: Finalized purchases. Relationship: ManyToOne with users.
  • order_items: Line items inside an order. Relationship: ManyToOne with shop_orders & products.
  • delivery_tracking: Live tracking status of an order. Relationship: OneToOne with shop_orders.

C. Spiritual & Cultural Services

  • acharyas: Profiles of priests/scholars.
  • acharya_poojas: Specific pooja services offered by priests. Relationship: ManyToOne with acharyas.
  • pooja_schedules: Global daily schedule of main rituals.
  • saints_directory: Directory of saints and Akhadas.
  • spiritual_places: Holy locations with coordinates and info.
  • heritage_history & kumbh_highlights: Static content for the informational portal.
  • artisans: Profiles of local craftsmen.

D. Stay & Accommodation

  • stays: Lodging options (tents, ashrams) with pricing and ratings.
  • stay_features: A dynamically generated collection table storing amenities (Wi-Fi, AC) for stays. Relationship: ManyToOne with stays (via @ElementCollection).

E. Public Services & Emergency (Smart City Operations)

  • live_updates: Broadcasts for general public info and essential services.
  • live_darshans: Streams and YouTube links for online viewing.
  • emergency_alerts: Geo-tagged SOS alerts raised by users (Pending/Accepted/Resolved).
  • hospitals & helplines: Directory of medical centers and emergency contacts.
  • health_tips: General health advisories for pilgrims.
  • crowd_status: Real-time tracking of crowd density (GREEN/YELLOW/RED) at specific coordinates.

F. Donations Module

  • donation_transactions: Records of user contributions, amounts, and payment references.
  • donation_configs: Admin-controlled UI configuration (UPI ID, target amounts, hero text) for the donation page.

3. System Data Flow

  • Authentication Flow: When a user registers/logs in, a User record is created/verified. The system generates an AccessToken and RefreshToken, saving them to the database mapping back to the user_id. When the access token expires, the client sends the refresh token to get a new access pair without requiring credentials.
  • E-commerce Flow:
    1. A user browses products and adds items.
    2. The system locates the user's Cart (or creates a 1:1 cart) and inserts cart_items referencing the cart and product.
    3. Upon checkout, a ShopOrder is generated. cart_items are migrated to order_items with a snapshot of the price. The cart is cleared.
    4. A DeliveryTracking record is linked 1:1 to the new ShopOrder for admin dispatch updates.
  • Donation Flow: The frontend fetches DonationConfig to dynamically render UPI details and preset amounts. User submits payment, creating a DonationTransaction. Admins later reconcile the "PENDING" status to "SUCCESS".
  • Stay Booking Flow: Users view Stays. The stay_features table is joined automatically by JPA to retrieve amenities (Wi-Fi, Meals).
  • Command Center Flow: The mobile app triggers an EmergencyAlert with GPS coordinates. The Admin/Operator dashboard polls this table. Once an operator clicks "Accept", the status changes from Pending to Accepted, updating the accepted_by field.

4. Special Technical Highlights

  • @ElementCollection vs @OneToMany: Instead of creating a separate full-blown Feature Entity to manage stay amenities, the project uses @ElementCollection in Stay.java. This creates the stay_features mapping table efficiently without the overhead of a separate primary key for the feature strings.
  • ddl-auto=update Behavior: Configured in application.properties, this tells Hibernate to compare the existing SQL schema with the Java @Entity classes on startup. It automatically generates new tables (like our 29 tables) or adds missing columns dynamically, saving the team from writing manual SQL DDL scripts.
  • Database-Backed Token Auth: Instead of pure stateless JWTs, this project persists AccessToken and RefreshToken entities. This is a highly secure design choice. It allows the admin to instantly revoke a specific user's access by deleting the token record from the database, which is impossible with standard stateless JWTs.
  • BaseEntity Auditing: Most tables extend BaseEntity, utilizing Spring Data JPA's @EntityListeners(AuditingEntityListener.class). This automatically injects createdAt and updatedAt timestamps on every insert and update across the system.

"Good morning, respected Professor and panel.

Our Full-Stack Maha Kumbh Mela project utilizes a highly relational and dynamic database architecture powered by Spring Boot Data JPA and MySQL. The system autonomously manages 29 distinct tables, generated directly from our backend Entity mappings using Hibernate's DDL update strategy.

To ensure separation of concerns, our schema is divided into 6 core modules: Authentication, E-commerce, Spiritual Services, Accommodation, Smart City Public Services, and Donations.

Let's look at the core relationships: Our security relies on a database-backed token system. A User has a strict one-to-one relationship with AccessToken and RefreshToken tables, allowing us to implement forced session revocations—a critical security feature for a high-traffic app.

For the E-commerce shop, we designed a robust checkout flow. A User maps one-to-one to a Cart, which holds one-to-many CartItems. Upon purchase, this transforms into a ShopOrder, linking one-to-many OrderItems, and triggering a one-to-one DeliveryTracking record. This guarantees data integrity so that past order prices aren't affected if the main Product price changes.

We also utilized advanced JPA features to optimize performance. For example, instead of creating a heavy entity for hotel amenities, our Stay entity uses an @ElementCollection. This seamlessly generates a lightweight stay_features mapping table managed entirely by Hibernate.

Finally, our Smart City module handles real-time CrowdStatus and EmergencyAlerts with geospatial coordinates, proving that our database isn't just for static data, but acts as the backbone for live operational dashboards.

Thank you, we are now open for any technical questions regarding the schema."

6. Viva Questions & Strong Answers

Q1: Why did you use spring.jpa.hibernate.ddl-auto=update? Is it safe for production?

"We used update to accelerate development, as Hibernate automatically syncs our Java @Entity models to the MySQL schema. While excellent for prototyping and adding columns on the fly without losing data, in a strict production environment, we would switch to a version-controlled migration tool like Liquibase or Flyway to ensure deterministic and reversible schema changes."

Q2: I see you have AccessToken and RefreshToken tables. Why store JWTs in the database? Doesn't that defeat the purpose of stateless JWTs?

"It is a deliberate security tradeoff. While JWTs are naturally stateless, storing them in the DB allows us to implement 'Token Revocation' and 'Single Device Login'. If an admin spots malicious activity, or a user logs out, we simply delete the token record. The backend intercepts the next request, checks the DB, and rejects the valid JWT—giving us stateful control over stateless tokens."

Q3: How are the amenities for the Stays (like Wi-Fi, AC) stored? I don't see a Feature entity.

"We implemented an @ElementCollection in the Stay class. This tells Hibernate to automatically create a separate stay_features table containing a stay_id foreign key and the feature string. It's much lighter and faster than creating a full @Entity for simple lists of strings, reducing unnecessary table joins."

Q4: How does your database track when a record was created or modified?

"Instead of writing redundant code in 28 files, we created an abstract BaseEntity mapped superclass. It uses Spring Data JPA's @EntityListeners(AuditingEntityListener.class) along with @CreatedDate and @LastModifiedDate annotations. Every entity that extends BaseEntity automatically receives and auto-updates the created_at and updated_at timestamps seamlessly."

Q5: In your E-commerce flow, how do you handle the situation where an item's price changes after a user has bought it? Will their old order show the new price?

"No, it won't. When a cart is checked out, we migrate CartItem data into OrderItem. The OrderItem table explicitly copies and stores the price at the moment of checkout. It holds a ManyToOne relationship with ShopOrder, acting as an immutable historical record, protecting past orders from future Product price fluctuations."
